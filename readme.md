# 卡牌游戏战斗系统设计文档

**风格**：炉石传说风格 | **架构**：FIFO 结算架构 | **实现**：GDScript (Godot)

---

## 一、 架构概览

系统采用 **命令队列 (Command Queue) + 原子操作 (Atomic Action) + 表现层事件分离** 的三层架构，确保逻辑计算与动画表现解耦。

**数据流向：**
`玩家输入` → `操作队列` → `原子操作` → `表现层（动画事件）`

### 核心原则

- **单线程结算**：任何时刻只有一个 Command 在结算，期间锁定玩家输入。
- **延迟计算**：伤害数值在原子操作执行时（Runtime）计算，而非分解指令时预计算。
- **统一死亡检查**：死亡检查在每个 Command 的所有原子操作完成后进行。
- **表现分离**：逻辑层计算完成后，一次性将动画事件列表发送给表现层。

---

## 二、 数据结构

### 2.1 枚举定义

| 枚举类型        | 值                                                                                                                        | 含义               |
| :-------------- | :------------------------------------------------------------------------------------------------------------------------ | :----------------- |
| **CommandType** | PLAY_MINION / PLAY_SPELL / MINION_ATTACK / HERO_ATTACK / HERO_POWER / END_TURN                                            | 玩家发起的各类操作 |
| **AtomType**    | DEAL_DAMAGE / RESTORE_HEALTH / SUMMON_MINION / DESTROY_MINION / TRIGGER_DEATHRATTLE / APPLY_BUFF / DRAW_CARD / SPEND_MANA | 原子操作类型       |
| **GameState**   | WAITING_INPUT / RESOLVING / GAME_OVER                                                                                     | 游戏状态与输入锁   |

### 2.2 核心类

- **Command (玩家指令)**：封装一次玩家操作（类型、发起者、目标、卡牌数据）。
- **AtomicAction (原子操作)**：指令分解后的最小逻辑单元（类型、目标、来源、数值、附加数据）。
- **AnimationEvent (表现层事件)**：原子操作执行后产生的动画指令（动画名、参数）。
- **GameEntity (游戏实体)**：随从与英雄的基类，包含 `hp`, `max_hp`, `attack` 等属性及 `take_damage` 等方法。

## 三、 完整结算流程

### 3.1 结算示例：随从 A 攻击随从 B

1. **输入采集**：UI 创建 `Command(MINION_ATTACK, A, B)` 并提交。
2. **状态校验**：检查 `GameState == WAITING_INPUT`，通过后入队并触发 `_process_queue()`。
3. **锁定输入**：状态切换至 `RESOLVING`。
4. **指令分解**：`_resolve_command()` 将攻击分解为 `[DEAL_DAMAGE(A→B), DEAL_DAMAGE(B→A)]`。
5. **原子执行**：
    - 执行 A 对 B 伤害：B 扣血，产生 `take_damage` 动画事件。
    - 执行 B 对 A 伤害：A 扣血，产生 `take_damage` 动画事件。
6. **死亡检查**：调用 `_check_deaths()`。若 B 死亡，触发其亡语并产生 `minion_death` 动画事件。
7. **动画推送**：`_flush_animations()` 将所有动画事件一次性发给表现层。
8. **解锁输入**：状态切回 `WAITING_INPUT`。

### 3.2 指令分解规则

- **MINION_ATTACK**：双向伤害原子操作。
- **PLAY_MINION**：扣费 + 召唤 + 战吼效果。
- **PLAY_SPELL**：扣费 + 法术效果。
- **END_TURN**：触发回合结束效果。

## 四、 关键设计决策

| 决策点           | 核心规则                                  | 理由                                                     |
| :--------------- | :---------------------------------------- | :------------------------------------------------------- |
| **伤害计算时机** | 执行时读取 `source.attack`                | 防止在同一结算帧内，因前序效果改变攻击力而导致数值错误。 |
| **死亡检查时机** | Command 所有原子操作执行后统一进行        | 实现「双方随从同时死亡」的效果；亡语需递归检查死亡。     |
| **输入锁**       | `RESOLVING` 状态拒绝一切 `submit_command` | 确保逻辑结算的原子性，防止操作冲突。                     |
| **同步策略**     | 全部算完再播                              | 逻辑层与表现层解耦，支持跳过动画，易于调试。             |

---

## 五、 扩展与同步

- **新卡牌**：仅需在 `_build_xxx_atoms()` 中返回原子序列，无需修改核心系统。
- **亡语系统**：由 `_check_deaths()` 自动调用实体的亡语方法，产生的原子操作会立即执行并递归检查死亡。
- **联机同步**：基于 Command 驱动。双端同步 Command 序列，本地执行 BattleSystem，表现层独立播放动画。
